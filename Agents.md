# Agent Guidelines

In all interactions, plans and commit messages, be extremely concise. Sacrifice grammar for the sake of concision.

At the end of each plan, list unresolved questions. Ask about edge cases, error handling, and unclear requirements before proceeding.
End every plan with a numbered list of concrete steps. This should be the last thing visible in the terminal.

Always use the most modern HTML, CSS, TypeScript, and React you can. If it's supported in every browser, even if it's only the most recent version, use it.

User should not have to ask you for your opinion explicitly. Always evaluate what the user is asking you to do, and voice your concerns before proceeding if you don't think it's a good idea. If possible, propose a better solution, but you can voice concerns even without one. This applies even to direct requests to revert or simplify. Still evaluate whether your original approach was better. The user may be missing important context. If there was a solid reasoning you suggested that approach, push back with reasoning instead of silently complying.

Interview me relentlessly about every aspect of this plan until we reach a shared understanding. Walk down each branch of the design tree, resolving dependencies between decisions one-by-one.

For any formatting and eslint issues ALWAYS RUN THE FOLLOWING COMMAND before manual fix:
`pnpm run check`

## Component Architecture

These rules apply to all features. They are enforceable and non-negotiable.

### Rules

1. **Atomic / SRP** — each component has a single responsibility. No god components.

2. **Independent** — no hidden external dependencies. All deps are explicit (props, hooks, imports).

3. **Barrel exports** — each feature exposes a single `index.ts` at its root. External code imports from `#/features/[name]` only, never from deep subpaths.
   - Internal files import each other directly (not through the barrel) to avoid circular deps.

4. **Feature-grouped** — all components, hooks, and utils for a feature live in `src/features/[name]/`.

5. **Common components** — globally reusable across 2+ features/routes go in `src/components/`. Criterion: used in 2+ places → common. Feature-specific → stays in feature.

6. **Build on shadcn/ui** — custom components extend shadcn primitives. Never reinvent what shadcn provides.

7. **Style with Tailwind** — `cn()` for conditional classes, `cva()` for component variants. No inline styles except dynamic values (e.g. `style={{ width: pct }}`).

8. **Routes are layout-only** — route files contain only: `createFileRoute`, search schema, and a component that composes feature components. No hooks, no state, no business logic. Loaders are the only exception.

9. **All feature logic in `src/features/`** — hooks, utils, types, sub-components. If it's logic, it does not belong in a route file.

10. **Small files** — ~100 lines is the signal to split. Internal sub-components always go in their own files, never co-located in a parent's file. One component per file.

11. **DRY** — shared UI used in 2+ places moves to `src/components/`. Shared logic moves to `src/hooks/`. Never duplicate.

12. **Orchestrator pattern** — parent owns all state + handlers, passes callbacks down to presentational children. Presentational components receive only what they need via props; no internal auth/query/state knowledge.

13. **Controlled components** — dialog/modal open state (`open`, `onCancel`, `onConfirm`) owned by the caller (orchestrator), never by the dialog itself.

### Feature Internal Structure

Flat by default. Add subfolders only when needed:

```
src/features/[name]/
├── index.ts                  ← barrel (public API of this feature)
├── [name]-component.tsx      ← kebab-case filenames
├── [name]-other.tsx
└── hooks/                    ← optional subfolder for feature-specific hooks
    └── use-[name].ts
```

### Hook Placement

- Feature-specific hook → `src/features/[name]/hooks/use-[name].ts`
- Cross-feature shared hook → `src/hooks/use-[name].ts`

### Example: Adding a Feature

```tsx
// src/features/auth/auth-sheet.tsx
export function AuthSheet({ open, onOpenChange }: AuthSheetProps) { ... }

// src/features/auth/index.ts
export { AuthSheet } from './auth-sheet'

// src/routes/__root.tsx — imports from barrel only
import { AuthSheet } from '#/features/auth'
```

## Code Style

### Imports

Group: builtin → external → internal (`#/`) → relative. Alphabetize within groups.

```tsx
import { useState } from 'react'

import { useQuery } from '@tanstack/react-query'

import { Button } from '#/components/ui/button'
import { cn } from '#/lib/utils'

import { FeatureComponent } from './feature-component'
```

Use `import type` for type-only imports (`@typescript-eslint/consistent-type-imports` enforced).

### JSX Patterns

- **Ternary over `&&`** (ESLint enforces via `@tanstack/eslint-config`)
- Sort props alphabetically, event handlers last

```tsx
// Correct
{
  isLoading ? <Spinner /> : null
}
;<Button
  className="..."
  disabled={loading}
  variant="primary"
  onClick={handleClick}
/>

// Wrong — will lint error
{
  isLoading && <Spinner />
}
```

### TypeScript

- Strict mode, no unused vars/params
- **Named exports only** — no `export default`
- Prefer inferring types; annotate explicitly only where inference fails

```tsx
// Correct
export { MyComponent }
export function myFunction() {}

// Wrong
export default MyComponent
```

### Formatting

- Tabs for indentation
- Single quotes
- Trailing commas
- Semicolons
- LF line endings

### Styling

```tsx
import { cva } from 'class-variance-authority'
import { cn } from '#/lib/utils'

const cardVariants = cva('base-classes', {
  variants: { size: { sm: 'p-4', lg: 'p-8' } },
  defaultVariants: { size: 'sm' },
})

<div className={cn(cardVariants({ size }), className)} />
```

## TanStack Router

### File-based Routes

```
src/routes/
├── __root.tsx         # Root shell (providers, Navbar, Toaster)
├── index.tsx          # /
├── search.tsx         # /search
├── watchlist.tsx      # /watchlist
└── movie.$id.tsx      # /movie/:id
```

### Route Definition (layout-only)

```tsx
import { createFileRoute } from '@tanstack/react-router'
import { z } from 'zod'

import { FeatureContent } from '#/features/[name]'

const searchSchema = z.object({ q: z.string().optional() })

export const Route = createFileRoute('/path')({
  validateSearch: searchSchema,
  component: PageComponent,
  async loader({ context }) {
    // Only place allowed to have logic in a route file
    await context.queryClient.prefetchQuery(...)
  },
})

function PageComponent() {
  // Zero hooks, zero state — compose feature components only
  return <FeatureContent />
}
```

## Auth

> Project-specific.

Firebase Auth via `src/integrations/auth/provider.tsx`.

```tsx
import { useAuth } from '#/integrations/auth/provider'

const { user, loading, signIn, signUp, signOut } = useAuth()
```

- `user` is `null` when unauthenticated, `FirebaseUser` when authenticated
- `loading` is `true` during initial auth state resolution — always guard UI behind `loading` before checking `user`
- Auth is per-component, not route-level — no `_protected` layout route

## Error Handling

- Use TanStack Query `isError` / `error` states for async errors
- Throw `Redirect` or `NotFoundError` from loaders (ESLint configured to allow these)
- Firebase errors expose a `code` string (e.g. `auth/user-not-found`) — handle explicitly, not generically

## Common Pitfalls

1. **Firebase auth timing**: `loading` is `true` on first render. Never assume `user === null` means unauthenticated until `loading` is `false`.
2. **Firestore rules**: Reads/writes fail silently in prod if security rules don't allow them. Test rules before deploying.
3. **TMDB rate limits**: 40 requests/10s per IP. Use `staleTime` generously; don't fetch on every keystroke without debouncing.
4. **No default exports**: Breaks tree shaking and import consistency. ESLint does not enforce this — discipline required.
5. **JSX conditionals**: Use ternary, never `&&`. `0 && <X />` renders `0`. ESLint enforces ternary.
6. **routeTree.gen.ts**: Never edit manually. It is regenerated by the Vite plugin on route file changes.
7. **Barrel depth**: Import from `#/features/[name]` only. Never `#/features/[name]/some-file`. Deep imports bypass the public API contract.
8. **`pnpm run check`**: Run this before committing. It writes Prettier + fixes ESLint. `pnpm run lint` and `pnpm run format` are read-only — they do not fix.
